Purpose

The purpose of this application is to allow group name references to be agnostic between SDLC instances.

Justification

In a ServiceNow SDLC environment which synchronizes ServiceNow user groups with other sources, such as Active Directory Integration, it is possible that groups of the same name can have different sys_id on each of the development, test and production instances. If a workflow is created on the development server with a Group Approval node, the node can fail when pushed to test and/or production. This requires the workflow to be modified on the test and production servers, to select the approval group again.

Complications of modifying workflows in Test/UAT and Production

A potential conflict can arise if a workflow is modified in Test/UAT and/or Production, and then the workflow needs to be updated later from the development instance. If any modification dates in the update set occur between the time when the update set was completed and the time when any modifications occur in the other instance, then it will become excessively cumbersome to get the new update set to succeed.

The Solution

This application allows a list of commonly known group names to be defined by the system administrator. The workflow developer can then utilize this solution to populate group approval nodes by group name, rather than by the sys_id.

Installation

  • Open the Group Name Registry repository website to get the actual URL for the repository
  • In ServiceNow, Navigate to Applications → Studio
  • Select Open from repository
  • Enter the URL from step one
